# DevOps 契约系统

作为开发者，我需要自定义工作规范，以便团队一致执行。

## 用户故事

### story 1：定义约束

作为开发者，我需要在Contract中声明约束，以便团队遵守。

```
场景：我想声明"禁止 force push 到 main"
验收：
- [ ] 可在 Contract 中添加新约束
- [ ] 约束格式为：禁止 <行为>
- [ ] 约束可被查询
```

### story 2：声明边界

作为开发者，我需要声明边界，以便明确什么能做、什么不能做。

```
场景：我想声明"何时用 feature 分支"
验收：
- [ ] 可声明边界条件和场景
- [ ] 边界格式为：何时 <条件>
```

### story 3：用例测试

作为开发者，我需要对约束进行测试，以验证约束有效性。

```
场景：我想测试"禁止 force push"约束
验收：
- [ ] 可创建测试场景
- [ ] 可执行测试
- [ ] 测试结果通过/失败
```

### story 4：检查约束

作为开发者，我需要在操作前检查是否违反约束。

```
场景：我想在 push 前检查是否违规
验收：
- [ ] 可在操作前运行检查
- [ ] 检查结果可阻止操作
- [ ] 有明确的错误提示
```

### story 5：生成文档

作为开发者，我需要从约束生成文档，以便团队查看。

```
场景：我想生成团队规范文档
验收：
- [ ] 可从约束生成 Markdown
- [ ] 文档包含全部约束和边界
- [ ] 文档可分享
```

---

## 验收标准

- [ ] 可添加/删除/修改约束
- [ ] 约束可被程序检查
- [ ] 检查可集成到工作流
- [ ] 文档可导出

---

## QA 测试

### TC 1：定义约束

#### TC 1.1：添加新约束

```
场景：在 Contract 中添加新约束
输入：禁止 force push 到 main
期望：
- 约束成功添加
- 可通过命令查询到约束
- 约束持久化到存储
```

#### TC 1.2：删除约束

```
场景：删除已有的约束
输入：约束 ID
期望：
- 约束成功删除
- 查询结果为空
```

#### TC 1.3：修改约束

```
场景：修改约束内容
输入：约束 ID，新内容
期望：
- 约束成功修改
- 修改后的约束可查询
```

### TC 2：声明边界

#### TC 2.1：添加边界

```
场景：声明边界条件
输入：何时用 feature 分支
期望：
- 边界成功添加
- 边界可被查询
- 边界包含条件和场景
```

#### TC 2.2：边界优先级

```
场景：多个边界条件重叠
输入：重叠的边界声明
期望：
- 可定义优先级
- 高优先级边界被优先匹配
```

### TC 3：用例测试

#### TC 3.1：约束触发测试

```
场景：测试"禁止 force push"约束
输入：模拟 git push --force 命令
期望：
- 测试检测到违规
- 测试返回失败
- 有清晰的错误信息
```

#### TC 3.2：边界触发测试

```
场景：测试"何时用 feature 分支"边界
输入：应使用 feature 分支的场景
期望：
- 边界返回通过
- 提示应使用 feature 分支
```

#### TC 3.3：边界拒绝测试

```
场景：测试边界拒绝
输入：不应使用 feature 分支的场景（如 hotfix）
期望：
- 边界返回拒绝
- 提示应使用 hotfix 分支
```

### TC 4：检查约束

#### TC 4.1：push 前检查

```
场景：在 git push 前运行约束检查
输入：git push 命令
期望：
- 检查运行并返回结果
- 如违规，阻止操作
- 提供错误提示
```

#### TC 4.2：多约束检查

```
场景：同时检查多个约束
输入：复杂操作场景
期望：
- 所有相关约束都被检查
- 返回所有违规项
- 全部通过才允许操作
```

#### TC 4.3：skip 检查

```
场景：紧急情况需要跳过检查
输入：--skip-check 参数
期望：
- 可选跳过检查
- 跳过记录到日志
- 需要确认
```
