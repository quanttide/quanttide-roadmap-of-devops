# release publish 改进路线图

> 2026-07-07 · 来自 v0.10.0 正式版发布复盘 + 本地 vs 云端边界分析

## 背景

v0.10.0 发布时，CHANGELOG 自动生成失败。更深层的问题是：devops-release 命令和 CLI 无法面对 AI 制造的海量不遵守流程随意跳过去的混乱。

## 双轨策略

### 本地 CLI 轨道

增量、一次性的操作，只需看当前工作区即可决定。

| 优先级 | 任务 | 工作量 | 说明 |
|--------|------|-------|------|
| P0 | ✅ 阶段晋级 CHANGELOG 占位 | 已完 | 无新提交时自动放占位符而非报错 |
| P1 | 🔲 正式版 CHANGELOG 从预发布条目汇总 | 中 | `get_comparison_tag` + `collect_prerelease_entries` + LLM 合并 |
| P2 | 🔲 release audit artifact 完整性矩阵 | 中 | 聚合检测 CHANGELOG/tag/Release 三者是否一致 |
| P3 | 🔲 release audit --fix | 小 | 对可自动修复的问题（如缺 Release）直接补充 |

### 云端轨道

需要全局视角、跨仓库协调、持久化状态的场景。

| 优先级 | 任务 | 工作量 | 说明 |
|--------|------|-------|------|
| P1 | 🔲 批量 artifact 完整性扫描 | 大 | 扫描全部 scope 的 CHANGELOG/tag/Release 一致性，输出不一致清单 |
| P2 | 🔲 批量自动修复 | 中 | 根据契约自动补充缺失 artifact（如缺 Release 的直接 create） |
| P2 | 🔲 跨仓库发布协调 | 大 | 子模块发布后自动更新父仓库 submodule 指针 + 关联仓库版本依赖 |
| P3 | 🔲 审计日志持久化 | 中 | 记录每次发布、每次绕过、每次修复的操作轨迹 |

## 分界线原则

| 判定条件 | 归属 |
|---------|------|
| 只需看当前工作区就能决定的 | 本地 CLI |
| 需要看所有 scope / 所有 tag / 所有 release 才能决定的 | 云端 |
| 只需一次操作就能修复的 | 本地 CLI |
| 需要批量收敛大量不一致的 | 云端 |
| 操作延迟敏感、需要开发者交互确认的 | 本地 CLI |
| 需要持久化存储操作记录的 | 云端 |
