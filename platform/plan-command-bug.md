## 问题描述

`qtcloud-devops plan` 命令的 `status` 和 `doctor` 子命令行为与预期差距较大。

## 具体问题

### 1. `plan doctor` 格式检查太弱

当前只检查三类格式细节：
- 版本头 v 前缀（`## [vX.Y.Z]` → `## [X.Y.Z]`）
- 分类标题大小写（`### added` → `### Added`）
- checkbox 空格（`-  [x]` → `- [x]`）

对于完全非标准的格式（如 `## P0 — 阻塞/错误` 作为版本头）既不报错也不修复，直接返回 `✅ 格式无误`。

### 2. `plan status` 对格式问题静默失败

当 ROADMAP.md 使用非标准版本头（如 `## P0 — 阻塞`）时，`status` 直接返回 `未找到规划条目`，不说明原因。

### 3. 架构缺陷：规则修复在前、LLM 在后

当前两个命令的流程都是"先规则后 LLM"：

```
doctor:   规则检测 → 发现无问题 → 跳过 LLM → 用户看到的 ✅ 格式无误
status:   规则解析 → 无法匹配 → 返回空 → 用户看到 未找到规划条目
```

应该反过来：**LLM 先判断，规则只做校验**。

## 实际用例

用户有以下 ROADMAP 内容：

```markdown
## P0 — 阻塞 / 错误

### 0.1 scripts/run_match.cmd 指向已删除的文件
- 引用 main.py（v0.5.0 已移至 src/__main__.py）

## P1 — 重要

### 1.1 提取器候选填充逻辑三重复制
```

执行 `plan doctor` → `✅ 格式无误`
执行 `plan status` → `未找到规划条目`

两个命令各自静默忽略，用户无法得知哪里出了问题。

## 解决方案

### 统一架构：LLM 先判断 → 规则校验

```
doctor:
  1. LLM 读取全文，理解非标准格式（## P0 — 阻塞），转换为标准格式
  2. 规则引擎校验 LLM 输出，修复细节问题（v 前缀、大小写、checkbox）
  3. 输出修复报告

status:
  1. 标准解析 → 有结果 → 直接输出
  2. 标准解析 → 无结果且有 ## 行 → LLM 尝试理解并转换 → 重试解析
  3. 规则校验：检查转换结果是否符合标准格式
```

### 实现要点

- `doctor_roadmap`：Phase 1 = LLM(全文), Phase 2 = `apply_rule_fixes`(结果)
- `doctor_llm` prompt 需明确：识别非标准 `##` 头、`###` 分类，转换为标准格式
- LLM 未配置时回退到纯规则修复（现状）
- `print_status_to` 在解析结果为空且文件含 `##` 时，触发 LLM 路径

### 格式规范文档化

- 在 `plan doctor --help` 和错误信息中输出标准格式说明
